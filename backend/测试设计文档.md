# 博客论坛系统 - 后端性能压测设计文档

> 适用架构：PostgreSQL (+pgvector) + Redis + RabbitMQ + Meilisearch + ByteMD  
> 压测目标：验证核心链路吞吐与延迟、识别性能瓶颈、确立容量基线、验证降级策略

---

## 一、压测目标与核心原则

### 1.1 压测目标
- **性能基线**：确立核心接口（Feed 拉取、点赞/收藏、混合搜索、热榜计算）的 P50/P95/P99 延迟与吞吐基准
- **瓶颈识别**：定位数据库连接池、Redis 内存、MQ 积压、向量索引等潜在瓶颈点
- **容量规划**：明确单机/集群可支撑的 QPS、并发用户数、数据量级，支撑扩容决策
- **降级验证**：验证缓存击穿、组件故障、流量洪峰等场景下的系统降级能力与恢复时间
- **回归防护**：建立压测自动化流水线，性能回退超过阈值自动阻断发布

### 1.2 核心原则
- **生产仿真**：压测数据、网络配置、中间件参数尽可能对齐生产环境
- **场景隔离**：不同业务场景独立压测，避免相互干扰导致结果失真
- **渐进加压**：从低并发逐步提升至目标值，观察系统拐点与雪崩阈值
- **指标量化**：所有压测结果必须量化输出，拒绝"感觉变快/变慢"的主观判断
- **可复现性**：压测脚本、数据、环境配置版本化，支持任意时间复现对比

## 二、压测场景矩阵

### 2.1 核心业务场景

| 场景名称 | 业务描述 | 并发用户数 | 持续时间 | 核心指标 | 通过阈值 |
|---------|---------|-----------|---------|---------|---------|
| **首页 Feed 拉取** | 用户打开首页，拉取关注作者的最新 20 篇文章 | 500 | 5 分钟 | P95 延迟、缓存命中率、单请求 DB 查询数 | P95 < 80ms、Hit > 90%、DB 查询/请求 < 3 |
| **文章详情页加载** | 用户点击文章，加载全文 + 作者信息 + 互动状态 | 800 | 5 分钟 | P95 延迟、Redis 命中率、静态资源加载时间 | P95 < 50ms、Meta 缓存命中 > 95%、首屏 < 1.5s |
| **点赞互动爆发** | 单篇爆款文章 1 分钟内收到 5000 次点赞 | 2000（集中单文章） | 2 分钟 | Redis QPS、MQ 积压长度、DB 行锁等待时间 | 行锁 < 15ms、积压 < 1000、无请求超时 |
| **收藏互动（含备注）** | 用户收藏文章并添加备注，验证同步事务性能 | 300 | 3 分钟 | 事务提交延迟、DB 连接池使用率、缓存删除耗时 | P95 < 30ms、连接池 < 70%、删除 < 1ms |
| **关注关系变更** | 用户关注/取关作者，验证缓存删除与重建 | 200 | 3 分钟 | 事务延迟、缓存删除成功率、下次拉取重建耗时 | 删除成功率 100%、重建 < 10ms、无脏数据 |
| **混合搜索（RRF）** | 用户搜索技术关键词，词法 + 语义融合排序 | 300 | 5 分钟 | 融合计算耗时、召回准确率、内存占用峰值 | P95 < 120ms、Top10 相关 > 85%、无 OOM |
| **搜索深度分页** | 用户翻页至第 5 页，验证降级策略 | 100 | 2 分钟 | 降级触发准确率、纯词法检索延迟、响应完整性 | 降级 100% 触发、延迟 < 80ms、结果完整 |
| **热榜定时计算** | 模拟 10 万文章规模，执行 24h 热榜计算任务 | 单任务并发 | 10 分钟周期 | 计算总耗时、ZSet 原子替换延迟、内存峰值 | 计算 < 3 秒、替换 < 50ms、内存 < 80% |
| **向量索引构建** | 新文章发布后异步生成语义向量并写入 pgvector | 50（后台任务） | 持续 10 分钟 | 单向量生成延迟、索引写入耗时、查询延迟影响 | 生成 < 200ms、写入 < 50ms、查询延迟增幅 < 10% |

### 2.2 极端场景（压力边界）

| 场景名称 | 触发条件 | 预期行为 | 验证指标 |
|---------|---------|---------|---------|
| **缓存雪崩** | 手动删除 500 个热点 `article:meta` Key 后突发流量 | 互斥锁重建生效，仅 1 次 DB 查询，其余等待 | DB 查询/请求 < 2、重建等待 < 100ms、无连接池耗尽 |
| **大 V 粉丝洪峰** | 头部作者 1 秒内新增 1000 粉丝 | 粉丝计数异步微批，行锁等待可控，无雪崩 | 行锁 < 20ms、微批聚合率 > 95%、计数 3 秒内同步 |
| **搜索词泛洪** | 同一关键词 1 秒内 500 次搜索请求 | RRF 融合缓存命中，避免重复计算 | 融合缓存命中 > 80%、重复计算 < 20%、延迟稳定 |
| **热榜计算冲突** | 热榜计算期间高频查询榜单接口 | ZSet 原子替换生效，无断层/混合状态返回 | 查询返回完整旧榜或新榜、无 50% 混合状态 |
| **向量检索长尾** | 输入 100 个低频技术词并发搜索 | pgvector HNSW 索引召回稳定，无超时 | 召回率 > 70%、P99 < 300ms、无索引扫描退化 |

### 3.2 数据准备
- **文章数据**：5 万篇，标题/摘要/正文长度符合生产分布，10% 为长文（>5000 字）
- **用户数据**：2 万用户，粉丝数符合幂律分布（1% 大 V 占 50% 粉丝）
- **互动数据**：点赞/收藏/关注关系按生产比例预置，爆款文章预置高互动量
- **向量数据**：5 万文章预生成 768 维向量，HNSW 索引构建完成
- **缓存预热**：热点文章元数据、关注列表、计数缓存预加载至 Redis

### 3.3 监控接入
- **应用层**：接口延迟分布、错误率、线程池使用率、GC 频率
- **数据库**：查询延迟、连接池使用率、行锁等待、慢查询数量
- **Redis**：内存使用率、命中率、命令延迟、Key 淘汰数量
- **MQ**：队列积压长度、消费速率、重试次数、死信数量
- **搜索引擎**：召回延迟、融合耗时、内存占用、索引碎片率
- **系统层**：CPU/内存/磁盘/网络使用率，容器资源限制

## 四、关键指标与阈值定义

### 4.1 延迟指标
| 指标 | 定义 | 计算方式 | 阈值要求 |
|------|------|---------|---------|
| **P50 延迟** | 50% 请求的响应时间 | 排序后取中位数 | 核心接口 < 30ms |
| **P95 延迟** | 95% 请求的响应时间 | 排序后取 95% 分位 | 核心接口 < 80ms |
| **P99 延迟** | 99% 请求的响应时间 | 排序后取 99% 分位 | 核心接口 < 200ms |
| **最大延迟** | 最慢请求的响应时间 | 取最大值 | < 500ms（超时前） |

### 4.2 吞吐指标
| 指标 | 定义 | 计算方式 | 阈值要求 |
|------|------|---------|---------|
| **QPS** | 每秒查询数 | 总请求数 / 持续时间 | 核心接口 ≥ 目标值 |
| **TPS** | 每秒事务数 | 成功事务数 / 持续时间 | 写接口 ≥ 目标值 |
| **并发用户数** | 同时发起请求的用户数 | 压测工具配置 | 支持目标并发无降级 |

### 4.3 资源指标
| 指标 | 定义 | 监控方式 | 阈值要求 |
|------|------|---------|---------|
| **CPU 使用率** | 应用/中间件进程 CPU 占用 | Prometheus node_exporter | < 80%（持续） |
| **内存使用率** | 应用/中间件内存占用 | Prometheus + Redis INFO | < 85%（触发告警） |
| **连接池使用率** | DB/Redis 连接池占用比例 | 应用埋点 + 中间件监控 | < 70%（预留缓冲） |
| **磁盘 IO** | 数据库写盘延迟与吞吐 | iostat + PG 慢日志 | 写延迟 < 10ms |

### 4.4 业务指标
| 指标 | 定义 | 计算方式 | 阈值要求 |
|------|------|---------|---------|
| **缓存命中率** | 请求命中缓存的比例 | 命中次数 / 总请求数 | 热点数据 > 95% |
| **降级触发率** | 触发降级策略的请求比例 | 降级次数 / 总请求数 | < 1%（正常场景） |
| **数据一致率** | 缓存与 DB 状态一致的比例 | 抽样校验一致数 / 总抽样数 | 100%（最终一致） |
| **错误率** | 请求返回 5xx/超时的比例 | 错误请求数 / 总请求数 | < 0.1% |

## 五、压测执行流程

### 5.1 准备阶段
1. **环境部署**：按配置要求拉起压测环境，验证各组件健康状态
2. **数据初始化**：导入预置数据，执行缓存预热，构建向量索引
3. **监控配置**：接入 Prometheus + Grafana，配置关键指标告警规则
4. **脚本验证**：执行小规模冒烟测试，验证压测脚本逻辑与参数正确性

### 5.2 执行阶段
1. **基线压测**：单场景低并发（10% 目标值）运行 1 分钟，记录基线指标
2. **渐进加压**：按 10%→30%→50%→80%→100%→120% 梯度增加并发，每级稳定 2 分钟
3. **稳态观测**：目标并发下持续运行 5-10 分钟，记录稳定期指标分布
4. **极限探索**：继续加压至系统拐点（延迟陡增/错误率上升），记录雪崩阈值
5. **恢复验证**：降低并发至正常水平，观察系统自动恢复能力与时间

### 5.3 极端场景专项
1. **缓存雪崩**：手动删除热点缓存后突发流量，验证互斥锁与降级策略
2. **组件故障**：模拟 Redis/MQ/搜索引擎不可用，验证降级路径与恢复时间
3. **数据倾斜**：集中请求单一大 V/爆款文章，验证热点隔离与限流效果
4. **网络抖动**：模拟 10-50ms 网络延迟，验证超时配置与重试策略

### 5.4 收尾阶段
1. **数据清理**：清除压测产生的临时数据，恢复环境至初始状态
2. **报告生成**：自动汇总指标数据，生成对比图表与瓶颈分析
3. **问题归档**：记录性能问题、优化建议、待验证假设，纳入迭代计划
4. **基线更新**：将本次压测结果归档为新版基线，供后续回归对比

## 六、结果分析与优化建议

### 6.1 瓶颈定位方法论
```
1. 延迟分析：按调用链拆解，定位耗时最长的组件/接口
2. 资源分析：关联延迟峰值与 CPU/内存/IO/网络指标，识别资源瓶颈
3. 队列分析：检查 MQ 积压、DB 连接池、Redis 管道，定位阻塞点
4. 代码分析：结合火焰图/慢查询日志，定位低效算法或冗余查询
5. 配置分析：对比生产配置，识别参数调优空间（如连接池大小、超时时间）
```

### 6.2 常见瓶颈与优化方向
| 瓶颈现象 | 可能原因 | 优化建议 |
|---------|---------|---------|
| Feed 拉取延迟高 | 多路归并算法复杂度高、缓存未命中、DB 查询未走索引 | 优化归并使用最小堆、预热关注列表缓存、添加复合索引 |
| 点赞计数不一致 | Lua 脚本逻辑错误、异步微批聚合丢失、幂等表失效 | 加强脚本单元测试、增加聚合校验、幂等表加唯一索引 |
| 搜索融合超时 | RRF 计算未缓存、向量检索未走索引、深度分页未降级 | 融合结果缓存 5 分钟、检查 HNSW 参数、强制分页降级 |
| 热榜计算慢 | 活跃集合查询未用 Pipeline、分数计算未向量化、ZSet 写入未批量 | 使用 Pipeline 批量查、SIMD 优化计算、批量 ZADD 替换 |
| 缓存击穿频繁 | 热点 Key 未加互斥锁、TTL 设置不合理、重建逻辑过重 | 增加重建锁、动态调整 TTL、精简重建字段 |
