# 博客论坛系统 - 完整架构设计文档

  

> 技术栈：PostgreSQL (+pgvector) + Redis + RabbitMQ + Meilisearch + ByteMD  

> 版本：v2.0 | 适用场景：硬核技术知识库 | 设计原则：轻量、可扩展、最终一致

## 一、系统架构总览

- **客户端层**：Web 端/App 端、ByteMD 编辑器、搜索页面，通过 HTTPS + JWT 与后端通信

- **网关层**：Nginx 负责路由转发、限流熔断、CORS 配置、静态资源缓存

- **应用服务层**：认证、文章、互动、搜索、向量五大模块独立部署，无状态设计支持水平扩展

- **数据与中间件层**：

  - PostgreSQL：核心关系数据持久化 + pgvector 扩展存储语义向量

  - Redis：实时去重集合、计数缓存、热榜有序集合、会话状态、活跃文章窗口

  - RabbitMQ：异步解耦文章发布、向量生成、统计落盘、活跃标记等耗时操作

  - Meilisearch：词法检索引擎，支持精确匹配、拼写容错、关键词高亮

  - 向量服务：抽象接口封装嵌入模型调用与向量存储检索

### 2.2 索引设计

- 用户表：username 唯一索引，status/role 普通索引

- 文章表：user_id、status、published_at 条件索引（仅已发布）、is_top 条件索引

- 统计表：hot_score_24h 降序条件索引（仅分数非空且文章已发布）

- 关注表：(follower_id, created_at DESC) 复合索引，支持拉模式增量查询

- 点赞/收藏表：(user_id, article_id) 唯一索引 + (article_id) 普通索引

- 向量表：pgvector HNSW 索引，支持余弦相似度快速检索

## 三、缓存设计（Redis）

### 3.1 Key 命名规范

格式：`{模块}:{子模块}:{标识符}:{字段}`  

示例：`auth:token:blacklist:{jti}`、`like:count:{articleId}`

### 3.2 核心缓存结构

| **模块**   | **Key 格式**                   | **类型** | **Value 说明**                      | **TTL/策略**             |
| -------- | ---------------------------- | ------ | --------------------------------- | ---------------------- |
| **认证**   | `auth:token:blk:{jti}`       | String | 黑名单标识 `1`                         | JWT 剩余有效期              |
| **认证**   | `auth:user:ban:{userId}`     | String | `reason\|expire_ts`               | 封禁时长或永久                |
| **用户**   | `user:meta:{id}`             | Hash   | 昵称、头像等信息                          | 2小时 + LRU              |
| **文章**   | `article:meta:{id}`          | Hash   | 标题、摘要、作者等信息                       | 2小时 + LRU              |
| **关注**   | `follow:uids:{userId}`       | Set    | 关注的作者 ID 列表                       | 10分钟                   |
| **关注**   | `follow:count:{userId}`      | String | 用户粉丝数                             | 10分钟                   |
| **拉模式**  | `feed:pull:ts:{userId}`      | String | 最后拉取时间戳                           | 10分钟                   |
| **互动**   | like:count:{articleId}       | String | 点赞总数                              | 10分钟                   |
| **互动**   | collect:count:{articleId}    | String | 点赞总数                              | 10分钟                   |
| **互动**   | `like:uids:{articleId}`      | Set    | 已点赞用户 ID 集合                       | 10分钟                   |
| **互动**   | `collect:uids:{userId}`      | Set    | 已收藏用户 ID 集合                       | 10分钟                   |
| 互动       | `comment:count:{articleId}`  | String | 评论总数                              | 10分钟                   |
| 互动       | `comment:recent:{articleId}` | ZSet   | 最新根评论 ID 列表（Score=`created_at`）   | 5 分钟，仅缓存 Top 20 根评论 ID |
| **热榜**   | `rank:hot:{24h/7d}`          | ZSet   | Member:articleId, Score:hot_score | 永久/定时重建                |
| **热榜**   | `rank:hot:cat:{cid}`         | ZSet   | 分类维度热榜                            | 永久/垂直榜单                |
| **搜索**   | `search:sugg:{prefix}`       | List   | 高频联想词列表                           | 1 小时                   |
| **活跃**   | `act:arts:win:{24h}`         | ZSet   | Member:articleId, Score:last_ts   | ZREMRANGE 清理           |
| **活跃**   | `act:arts:win:{7d}`          | ZSet   | Member:articleId, Score:last_ts   | ZREMRANGE 清理           |
| **Feed** | `feed:auth:ids:{id}`         | ZSet   | Member:articleId, Score:pub_at    | 用于快速拉取                 |
缓存失效从DB回填时使用 **Singleflight 机制**
## 四、消息队列设计（RabbitMQ）

### 4.1 拓扑结构

- Exchange：`article.events`（Direct 类型），负责文章相关事件路由

- 队列 1：`search.indexer`，绑定 key=`article.*`，消费方：Meilisearch 同步服务

- 队列 2：`embedding.generator`，绑定 key=`article.published`，消费方：向量生成服务

- 队列 3：`stat.sync`，绑定 key=`stat.update`，消费方：统计落盘服务

- 队列 4：`activity.tracker`，绑定 key=`interaction.*`，消费方：活跃集合标记服务

- 死信队列：`dlx.article`，处理同步失败消息，支持重试与告警

### 4.2 消息体规范

**文章事件消息**

- event_id：UUID v4，全局唯一标识
- event_type：article.published / article.updated / article.deleted
- timestamp：事件发生时间（ISO 8601 格式）
- payload：article_id、user_id、title、summary、content_md、tags、category_ids、published_at、cover_img

**互动事件消息**

- event_id：UUID v4
- event_type：like.added / like.removed / collect.added / collect.removed
- timestamp：事件时间
- payload：user_id、article_id、action（add/remove）

**向量生成任务**

- event_id：UUID v4
- article_id：待处理文章 ID
- text_to_embed：标题 + 摘要 + 正文前 2000 字符的纯文本拼接
- model_version：嵌入模型版本标识
## 五、搜索引擎设计（词法 + 语义双引擎）

### 5.1 词法检索（Meilisearch）

- 可搜索字段：title、summary、content_text（Markdown 转纯文本）、tags
- 可过滤字段：user_id、category_ids、tags、status、published_at
- 可排序字段：published_at、like_count、collect_count、hot_score
- 排序规则：词数匹配 → 拼写容错 → 字段邻近度 → 属性权重 → 精确匹配 → 自定义排序
- 特性配置：同义词映射（技术术语）、停用词过滤（中文虚词）、拼写容错阈值、关键词高亮标签
### 5.2 语义检索（pgvector）

- 嵌入模型：统一使用sentence-transformers，输出 768/1024 维向量
- 存储方案：PostgreSQL + pgvector 扩展，向量与业务数据同库同事务
- 索引类型：HNSW 索引，距离度量使用余弦相似度
- 更新策略：文章发布/修改时异步触发向量生成，版本控制支持模型平滑升级
- 查询接口：标准 SQL 支持 `ORDER BY embedding <=> query_vector`，可结合 WHERE 条件过滤
### 5.3 混合检索策略（RRF 融合）

- 召回阶段：Meilisearch 返回 Top 100 词法结果，pgvector 返回 Top 100 语义结果
- 融合阶段：使用倒数排名融合公式 `Score = Σ 1/(k + rank)`，k 默认取 60
- 权重调节：支持前端切换"精确优先/语义优先"，后台动态调整融合系数
- 过滤前置：状态、分类、时间等硬过滤条件在双引擎查询时同步下发，减少无效召回
## 六、核心业务流程

### 用户登录与鉴权

#### 阶段一：登录签发（低频操作）

- 客户端提交 username + password
- 服务端查询数据库校验凭证 Bcrypt 与账号 status
- 校验通过：
    - 生成 **Access Token**：有效期 30 分钟，Payload 仅含 `userId`、`role`、`exp`
    - 生成 **Refresh Token**：有效期 7 天，Payload 含 `userId`、`jti`（唯一标识）
    - 初始化 Redis 会话：写入 `auth:session:{userId}`，TTL 设为 7 天
- 返回响应：Access Token 交由客户端内存/Header 管理；Refresh Token 写入 `HttpOnly Secure Cookie` 防 XSS 窃取
#### 阶段二：日常业务请求（热路径·真无状态）

- 客户端请求携带 `Authorization: Bearer <access_token>`
- 网关/拦截器执行：
    - 解析 JWT，校验签名算法与 `exp` 过期时间
    - 不查询 Redis，不更新会话，不校验黑名单
- 校验通过 → 注入 `userId/role` 到请求上下文 → 直接放行
- 设计权衡：接受最长 30 分钟的封禁/登出延迟，换取零 Redis IO 的极致网关性能
#### 阶段三：Token 刷新（冷路径·状态校验与会话保活）

- Access Token 过期返回 401 → 客户端自动调用 `POST /auth/refresh`，携带 Cookie 中的 `refresh_token`
- 服务端验签 Refresh Token，提取 `userId` 与 `jti`
- **此时才查询 Redis**：
    - 查 `auth:session:{userId}` 是否存在且未过期
    - 查 `auth:user:ban:{userId}` 是否命中封禁标记
    - 查 `auth:token:blacklist:{jti}` 是否已主动登出
- 校验通过 → 签发全新 Access Token → 刷新 Redis `auth:session` TTL 为 7 天 → 返回新 Token 对
- 校验失败 → 返回 401/403，客户端清除本地 Token 并跳转登录页
### 阶段四：登出与封禁处理

- **主动登出**：客户端调用登出接口 → 后端将当前 `jti` 加入 Redis 黑名单（TTL = 该 Token 剩余有效期）→ 清理 `auth:session` → 客户端丢弃双 Token
- **后台封禁**：管理员操作 → Redis 写入 `auth:user:ban:{userId}` → 用户下一次尝试刷新 Token 时被拦截（Access Token 最长残留 30 分钟）
- **强制踢人**：逻辑同封禁，仅标记状态，不修改角色权限
### 文章模块

### 文章发布
- 客户端提交文章数据（title、content_md、cover_img、tags、category_ids）
- 服务端 JWT 鉴权 + 参数校验 + XSS 过滤（白名单库）
- 写入 PostgreSQL：sys_article + sys_article_stat（初始计数为 0）
- 文章元数据写入缓存
- 发布文章事件至 RabbitMQ：article.events（event_type=article.published）
- 搜索消费者异步消费：Markdown 转纯文本 + 分词预处理 + 写入 Meilisearch
- 向量消费者异步消费：文本分块 + 调用嵌入模型 + 写入 pgvector 表
- 返回响应：201 Created + article_id
#### 获取文章细节
- `HMGET article:meta:{id}` → 拿静态字段  `GET like:count:{id}` + `SISMEMBER like:uids:{id} {userId}` → 拿动态状态/计数，没有则查DB回填。
- 查DB获取全文
### 互动模块
### 点赞

- 客户端发起点赞/取消点赞请求
- 服务端执行核心 Lua 脚本：原子校验+修改计数器
- 修改成功则推至 RabbitMQ（仅含 `article_id`, `user_id`, `event_id`）
- 主流程同步返回结果：`{liked: true/false, count: 156}`
- 统计消费者：接收消息后执行**内存微批聚合**，彻底规避爆款文章行锁竞争
- 活跃标记消费者：微批聚合后，独立 Pipeline 批量 `ZADD` 至 `act:articles:window:*`；由独立定时任务每 10 分钟执行 `ZREMRANGEBYSCORE` 清理过期窗口
- 全链路基于 `event_id` 实现消费幂等（`sys_event_log` 表 `ON CONFLICT DO NOTHING`）
### 收藏

- 客户端发起收藏/取消收藏请求
- 落库（修改关系表和计数表）
- 同步删除 Redis
- 异步旁路：发布轻量事件至 MQ，仅用于更新活跃集合
### 关注

- 客户端发起关注/取关请求
- 落库
- 删除缓存
#### 评论
- 客户端提交根评论/回复
- 落库
- 异步旁路：发布轻量事件至 MQ，仅用于更新活跃集合
- **查询链路**：
	- 根评论：`WHERE article_id=? AND parent_id=0 AND status=1 ORDER BY created_at DESC`
	- 回复列表：`WHERE parent_id=? AND status=1 ORDER BY created_at ASC`
	- 首屏加速：Redis `ZRANGE comment:recent:{id} 0 19` 获取根评论 ID → Pipeline 查 DB 详情 → 渲染
### 文章列表获取
### 拉模式：获取关注动态

- 客户端请求 `/api/feed/following?cursor=&limit=20`
- 服务端解析 cursor，无则取默认起始时间（如 24 小时前或 `feed:last_pull` 时间）
- 并发查询各关注作者的 `feed:author:ids:{authorId}` ZSet，执行 `ZREVRANGEBYSCORE` 获取时间大于等于游标时间的候选 ID 集合
- **应用层多路归并**：
    - 将所有候选 ID 按 `(published_at DESC, id DESC)` 严格排序
    - 执行复合游标过滤，剔除游标位置及之前的记录
    - 截取前 `limit` 条作为本页结果
- Redis Pipeline 批量预取元数据和互动状态，DB 批量查询缺口数据，回填缓存
- 组装响应，生成下一页 cursor：取结果集最后一条的 `(time, id)` 拼接返回
- 更新 `feed:last_pull:{userId}` 为本次查询的最早时间戳，供静默刷新使用
### 混合搜索查询

- 客户端提交搜索请求（关键词 + 分类过滤 + 排序偏好 + 分页）
- 服务端并行请求 Meilisearch（词法）与 pgvector（语义）
- 双引擎按 RRF 公式融合打分，去重后截取 Top N
- Redis Pipeline 批量预取元数据和互动状态，DB 批量查询缺口数据，回填缓存。
- 组装高亮片段与排序结果返回前端
### 热榜计算与榜单更新

- 定时任务触发：24h 热榜每 10 分钟，7d 热榜每小时
- 构建计算池：
  - 步骤 1：ZRANGEBYSCORE 获取 act:articles:window:24h 中最近 24h 活跃文章 ID

## 七、部署与运维

### 基础设施要求

- PostgreSQL 15+：启用连接池、慢查询日志、自动 vacuum，安装 pgvector 扩展

- Redis 7+：配置 maxmemory=512mb、maxmemory-policy=allkeys-lru、开启 AOF 持久化

- RabbitMQ 3.12+：配置镜像队列、消息持久化、死信交换机

- Meilisearch 1.5+：生产环境启用主键认证、定期快照备份

- 应用服务：无状态部署、健康检查接口、支持水平扩展

### 监控指标

- 认证模块：登录成功率、Token 校验耗时、黑名单命中率

- 互动模块：点赞/收藏成功率、缓存命中率、统计同步延迟

- 搜索模块：词法召回率、语义检索延迟、融合查询总耗时

- 热榜模块：计算任务耗时、活跃集合写入延迟、榜单更新成功率
