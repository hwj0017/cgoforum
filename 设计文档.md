# 博客论坛系统 - 完整架构设计文档（精简版）

> 技术栈：PostgreSQL + Redis + RabbitMQ + Meilisearch + ByteMD

---

## 一、系统架构总览

- **客户端层**：Web/App、ByteMD 编辑器、搜索页面，通过 HTTPS + JWT 与后端通信
- **网关层**：Nginx 负责路由转发、限流、CORS、静态资源缓存
- **应用服务层**：认证、文章、互动、搜索四大模块独立部署，无状态设计支持水平扩展
- **数据层**：
  - PostgreSQL：核心业务数据持久化
  - Redis：高频读写缓存、实时计数、热榜排序
  - RabbitMQ：异步解耦、事件驱动、最终一致性保障
  - Meilisearch：全文检索、智能排序、关键词高亮

### 2.2 索引设计

- **用户表**：username 唯一索引，status/role 普通索引

- **统计表**：hot_score_24h 降序索引（条件：分数非空且文章已发布）
- **互动表**：(article_id, action_type)、(user_id, action_type) 复合索引
- **标签关联表**：tag_id 索引，支持按标签反查文章
- **同步日志表**：sync_status=0 的部分索引（待处理任务），article_id 索引

### 2.3 优化配置

- 启用 pg_stat_statements 扩展监控慢查询
- 使用触发器自动维护 updated_at 字段
- 大文本字段（content_md）使用 TOAST 压缩存储
- 高频查询字段添加覆盖索引，减少回表

## 三、缓存设计（Redis）

### 3.1 Key 命名规范

格式：`{模块}:{子模块}:{标识符}:{字段}`
示例：`auth:token:blacklist:{jti}`、`act:like:count:{articleId}`

### 3.2 核心缓存结构

| 模块 | Key 格式 | 数据类型 | Value 说明 | TTL/策略 |
|------|---------|---------|-----------|---------|
| 认证 | auth:token:blacklist:{jti} | String | 1 | JWT 剩余有效期 | 用户登出时设置 |
| 认证 | auth:user:ban:{userId} | String | ban_reason\|expire_timestamp | 封禁时长或永久 | 管理员设置 |
| 认证 | auth:session:{userId} | String | last_active_timestamp | 30 分钟滑动 | 每次请求刷新 |
| 互动 | act:like:users:{articleId} | Set | 已点赞用户 ID 列表 | 永久（文章删除时清理） | 判断是否已点赞 |
| 互动 | act:like:count:{articleId} | String | 点赞总数 | 永久 | 实时展示，异步落盘 |
| 互动 | act:collect:users:{articleId} | Set | 已收藏用户 ID 列表 | 永久 | 判断是否已收藏 |
| 互动 | act:collect:count:{articleId} | String | 收藏总数 | 永久 | 实时展示，异步落盘 |
| 互动 | act:view:incr:{articleId} | Hash | timestamp/count | 10 分钟 | 本地合并后批量刷入 |
| 热榜 | rank:hot:24h | ZSet | Member:articleId, Score:hot_score | 永久（定时重建） | 前端直接读取 |
| 热榜 | rank:hot:7d | ZSet | 同上 | 永久 | 同上 |
| 热榜 | rank:hot:cat:{cid}:24h | ZSet | 分类维度热榜 | 永久 | 垂直领域榜单 |
| 搜索 | search:suggest:prefix:{kw} | List | 高频联想词列表 | 1 小时 | 前端输入联想 |
| 缓冲 | buffer:like:flush:{workerId} | Hash | articleId:delta | 5 秒 | 应用层本地合并 |

### 3.3 原子操作保障

- 点赞/取消操作使用 Lua 脚本保证去重集合与计数器原子更新
- 脚本逻辑：检查是否已操作 → 未操作则添加集合 + 计数增加 → 已操作则移除集合 + 计数减少 → 返回结果与当前计数
- 批量刷入使用 Redis Pipeline 减少网络往返，提升吞吐量

## 四、消息队列设计（RabbitMQ）

### 4.1 拓扑结构

- Exchange：article.events（Direct 类型）
- 队列 1：search.indexer，绑定 key=article.*，消费方：Meilisearch 同步服务
- 队列 2：stat.sync，绑定 key=stat.update，消费方：统计落盘服务
- 死信队列：dlx.article，处理同步失败消息，支持重试与告警

### 4.2 消息体规范

**文章事件消息（article.events）**
- event_id：UUID v4，全局唯一
- event_type：article.published / article.updated / article.deleted
- timestamp：事件发生时间（ISO 8601）
- payload：article_id、user_id、title、summary、content_md、tags、category_ids、published_at、cover_img
- metadata：retry_count、source_service

**互动统计消息（stat.events）**
- event_id：UUID v4
- event_type：interaction.batch_sync
- timestamp：批次生成时间
- payload：batch_id、items 数组（article_id + 各互动类型增量 + 同步时间）

**搜索同步确认消息（search.ack）**
- event_id：UUID v4
- correlation_id：关联原始事件 id
- status：success / failed
- article_id：处理的文章 id
- error_msg：失败时的错误信息
- processed_at：处理完成时间

### 4.3 可靠性配置

- 生产者开启发布确认（publisher-confirm-type=correlated）
- 消费者手动 ACK，业务处理成功后再确认消费
- 队列设置持久化（durable=true），防止重启丢失
- 消息设置 TTL（24 小时），避免无限堆积
- 预取数量：搜索任务 10，统计任务 50，平衡吞吐与内存

## 五、搜索引擎设计（Meilisearch）

### 5.1 索引配置

- primaryKey：article_id
- searchableAttributes：title、summary、content_text（Markdown 转纯文本）、tags
- filterableAttributes：user_id、category_ids、tags、status、published_at
- sortableAttributes：published_at、view_count、like_count、hot_score
- rankingRules：words → typo → proximity → attribute → exactness → custom_ranking(title_match_score desc) → custom_ranking(published_at desc)
- synonyms：技术术语映射（pg→postgresql、js→javascript）
- stopWords：中文虚词过滤（的、了、在、是、我、有、和）
- typoTolerance：启用拼写容错，最小词长 5 容 1 错、9 容 2 错

### 5.2 文档结构

- 核心字段：article_id、user_id、title、summary、content_text、tags、category_ids、status、published_at、cover_img
- 统计字段：view_count、like_count、collect_count、hot_score
- 高亮字段：formatted.title、formatted.summary
- 匹配分字段：title_match_score（标题匹配权重，用于自定义排序）

### 5.3 查询策略

- 支持关键词搜索 + 分类/标签/状态/时间过滤 + 多字段排序
- 返回字段精简：仅包含 article_id、title、summary、cover_img、published_at、user_id
- 分页：limit/offset 模式，前端配合无限滚动
- 高亮：自定义标签，前端样式控制
- 搜索建议：前缀匹配 + 高频词缓存 + 搜索引擎 suggest API 兜底

## 六、核心业务流程

### 6.1 用户登录与鉴权

- 客户端提交 username + password
- 服务端查询数据库校验凭证与账号状态
- 校验通过生成 JWT（Payload：userId、role、jti、exp），RS256 签名
- 客户端存储 Token，后续请求携带 Authorization: Bearer {token}
- 网关解析 JWT，校验签名与过期时间
- 查询 Redis 黑名单：拦截已登出（jti 黑名单）或封禁用户（userId 黑名单）
- 校验通过注入 userId/role 到请求上下文，放行请求

### 6.2 文章发布与索引同步

- 客户端提交文章数据（title、content_md、cover_img、tags、category_ids）
- 服务端 JWT 鉴权 + 参数校验 + XSS 过滤（bluemonday/DOMPurify）
- 写入 PostgreSQL：sys_article + sys_article_stat（初始计数为 0）
- 发布事件至 RabbitMQ：article.events（event_type=article.published）
- 搜索消费者异步消费：Markdown 转纯文本 + 分词预处理 + 写入 Meilisearch
- 统计消费者预留扩展：后续支持互动分析、用户画像等
- 返回响应：201 Created + article_id

### 6.3 点赞互动与异步落盘

- 客户端发起点赞请求（POST /articles/{id}/like）
- 服务端执行 Lua 脚本：检查去重集合 → 未点赞则 SADD + INCR → 已点赞则 SREM + DECR
- 实时返回结果：liked=true/false + 当前计数
- 同时更新应用层本地缓冲：ConcurrentHashMap 累计增量
- 触发条件：单文章累计 50 次 或 距离上次刷入超 2 秒
- 批量刷入：Pipeline 更新 Redis Hash 计数 + 发布 stat.events 至 RabbitMQ
- 统计消费者异步持久化：UPSERT sys_article_stat，更新 like_count/collect_count

### 6.4 热榜计算与榜单更新

- 定时任务触发：24h 热榜每 10 分钟，7d 热榜每小时
- 构建计算池：合并近期活跃文章 ID（Redis Set）+ 上一轮热榜 Top 200（Redis ZSet）
- 批量获取数据：从 PostgreSQL 拉取候选文章的互动统计 + 发布时间
- 内存计算热度分：代入加权衰减公式（收藏×10 + 评论×5 + 点赞×1 + 浏览×0.1）/ (小时数+2)^1.2 × 信任系数
- 阈值过滤：新旧分数差异<0.5% 则跳过，减少无效写入
- 原子更新榜单：先写临时 ZSet → 校验无误 → RENAME 替换旧榜单
- 异步持久化：更新 PostgreSQL 的 hot_score_24h/hot_score_7d 字段，支持降级查询

### 6.5 搜索查询与结果返回

- 客户端提交搜索请求（关键词 + 分类过滤 + 排序方式 + 分页）
- 服务端解析参数，调用 Meilisearch 执行检索
- 搜索引擎返回：匹配文档 ID、标题、摘要、高亮片段、匹配分
- 服务端批量查询数据库：根据 ID 获取完整文章信息（作者、封面、发布时间）
- 组装响应：合并搜索结果与文章详情，返回前端
- 搜索建议接口：优先读取 Redis 缓存的高频联想词，无缓存时回源查询 Meilisearch suggest API 并回填缓存（1 小时 TTL）

## 七、安全与合规

### 7.1 关键安全措施

| 风险点 | 防护措施 | 实现位置 |
|--------|---------|---------|
| XSS 攻击 | 后端白名单过滤 + 前端 CSP | Article Service + Nginx |
| SQL 注入 | 参数化查询 + ORM 预编译 | 所有数据库操作 |
| JWT 伪造 | RS256 非对称签名 + 私钥服务端保管 + jti 黑名单 | Auth Service |
| 接口越权 | 拦截器校验资源所有者 + RBAC 注解 | Gateway/Interceptor |
| 刷赞刷榜 | Redis 限流 + 用户信任分 + 异常行为检测 | Interaction Service |
| 恶意上传 | 文件类型校验 + 大小限制 + OSS 回调签名 | Upload Service |
| 搜索注入 | 参数白名单 + 关键词长度限制 | Search Service |

### 7.2 数据合规

- 用户注销时执行匿名化：用户名/密码/昵称脱敏，状态标记为已注销
- 文章内容保留但作者信息脱敏，平衡内容价值与隐私保护
- 互动关系保留用于统计，移除个人标识字段
- 搜索索引同步删除敏感字段，确保检索结果合规
- 提供用户数据导出接口，满足个人信息保护法要求

## 八、部署与运维

### 8.1 基础设施要求

- PostgreSQL 15+：启用连接池、慢查询日志、自动 vacuum
- Redis 7+：配置 maxmemory=512mb、maxmemory-policy=allkeys-lru、开启 AOF 持久化
- RabbitMQ 3.12+：配置镜像队列、消息持久化、死信交换机
- Meilisearch 1.5+：生产环境启用主键认证、定期快照备份
- 应用服务：无状态部署、健康检查接口、支持水平扩展

### 8.2 监控指标

- 认证模块：登录成功率、Token 校验耗时、黑名单命中率
- 文章模块：发布接口 P99 延迟、内容过滤耗时、数据库写入耗时
- 互动模块：点赞/收藏成功率、缓存命中率、批量刷入频率
- 搜索模块：查询延迟、召回率、高亮渲染耗时
- 基础设施：数据库连接数、Redis 内存使用率、队列积压长度、搜索引擎健康状态

### 8.3 告警策略

- 热榜计算失败或延迟>3s 触发警告
- Redis 内存使用率>80% 触发临界告警
- 消息队列积压>1000 条触发扩容建议
- 数据库慢查询占比>5% 触发优化提醒
- 搜索引擎健康检查失败触发紧急告警
## 九、设计原则总结

1. **读写分离**：高频读走缓存，高频写走异步，核心数据最终一致
2. **渐进增强**：先跑通最小闭环，再按需引入复杂组件
3. **降级兜底**：每个外部依赖都有备用方案，确保核心链路可用
4. **可观测性**：关键路径埋点，异常快速定位，容量提前规划
5. **安全左移**：权限校验前置，输入过滤早期，合规设计内建
